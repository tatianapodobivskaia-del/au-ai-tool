From Folder to Live App

You'll turn a folder on your laptop into a versioned Git project, push it to GitHub, and deploy it as a live website — all from the terminal. By the end, your AI tool will have a public URL anyone can visit.

⏱ 12 steps · ~2 hours 🎯 Your own live AI app 💻 Terminal-first
How this works. Each step has four parts: What to do (numbered instructions), the exact terminal commands to type (copy them character-for-character), a ✓ Checkpoint (what you should see on success), and a Stuck? box with the most common failure modes. Your instructor will demo each step on the projector first — then you do it. If you're ever ahead or behind, the step numbers sync up with what's on screen.

Phase 1 · Local
1

Download the Starter Files

~3 min

Grab the ZIP your instructor shared, unzip it to your Desktop, and rename the folder. It's just files right now — no version control yet. That comes next.

What to do

  1. Click the ZIP link your instructor shared (in the class chat / Moodle / email).
  2. The file au-ai-tool-starter.zip downloads to your Downloads folder.
  3. Move the ZIP to your Desktop, then double-click it to unzip.
  4. You now have a folder called au-ai-tool-starter on your Desktop.
  5. Rename it to something personal — your name, the tool's name, whatever. Example: my-ai-tool or socratic-tutor.
✓
There's a folder on your Desktop (with your chosen name) containing index.html, README.md, .gitignore, and LICENSE.
Stuck?

"I don't see the .gitignore file" → it's hidden. On Mac: ⌘+⇧+. in Finder. On Windows: View → Show → Hidden items.

"The ZIP didn't extract" → right-click it → Open With → Archive Utility (Mac) or Extract All (Windows).

2

Open a Terminal in Your Folder

~2 min

For the whole rest of the workshop, you'll work from the terminal — inside the folder you just renamed. Getting the terminal to "live" in the right place is step one.

What to do

  1. Open VS Code.
  2. File → Open Folder → select your renamed folder on the Desktop → Open.
  3. Open VS Code's built-in terminal: Ctrl+` (that's the backtick key, under Esc).
  4. A terminal panel opens at the bottom, already inside your folder.

Verify you're in the right place

confirm your current working directory pwd /Users/yourname/Desktop/my-ai-tool list the files — you should see your starter files ls LICENSE README.md index.html
✓
pwd shows a path ending in your folder's name, and ls shows index.html among the files. .gitignore is hidden by default — that's fine, ls -a would show it.
Stuck?

Windows PowerShell shows different paths → that's fine. Use forward slashes in commands; PowerShell accepts them.

"pwd: command not found" → you're in Windows Command Prompt instead of PowerShell. Close it, open PowerShell instead.

3

Initialize the Repo with git init

~3 min

Right now your folder is just files. After this one command, it becomes a Git repository — a tracked project with a full history waiting to be filled. This is the single most important command in Git.

Run this

turn this folder into a Git repository git init -b main Initialized empty Git repository in /Users/.../my-ai-tool/.git/

The -b main part tells Git to call the default branch main (instead of the older master). It's a one-time setting per repo.

See what just happened

list all files including hidden ones ls -a . .git .gitignore README.md .. LICENSE index.html

That new .git folder is where every commit, every branch, every version of every file will live from now on. If you ever delete .git, you lose all Git history — the files stay, but the "project" is gone.

✓
ls -a shows a .git directory in your folder. Your folder is now a Git repository.
🎉 First threshold

That one command — git init — is how every project on GitHub, on Kaggle, on Hugging Face, inside OpenAI, inside Anthropic, inside every AI company you know, begins. You just did it.

4

Tell Git Who You Are

~2 min

Every commit gets stamped with your name and email. Do this once per laptop — the settings are saved globally.

Run these (with your own info)

replace with your real name and email git config --global user.name "Your Name" git config --global user.email "you@example.com" confirm they saved git config --global --list user.name=Your Name user.email=you@example.com

Use the same email you signed up for GitHub with — that way your commits link to your profile.

✓
git config --global --list shows your name and email.
Stuck?

"I don't know which email my GitHub uses" → go to github.com/settings/emails. Use that one.

5

Stage & Make Your First Commit

~6 min

Committing is a two-step motion: stage (tell Git which changes to save) and then commit (save them with a message). Your first commit captures everything — the starting point of your project's history.

See what Git sees

git status On branch main No commits yet Untracked files: (use "git add <file>..." to include in what will be committed) .gitignore LICENSE README.md index.html

Git sees the files but isn't tracking them yet. "Untracked" means "I know you exist, but I haven't agreed to remember you."

Stage everything, then commit

stage all files (the dot means "everything in this folder") git add . confirm they moved to "staged" git status Changes to be committed: new file: .gitignore new file: LICENSE new file: README.md new file: index.html commit them with a short, present-tense message git commit -m "Initial commit — AI tool starter" [main (root-commit) a7f3c2b] Initial commit — AI tool starter 4 files changed, 928 insertions(+)

See your history

git log --oneline a7f3c2b (HEAD -> main) Initial commit — AI tool starter
✓
git log --oneline shows one commit on branch main. Your project officially has a history.
◆ Quick anatomy
What you just saw
a7f3c2b
The first 7 characters of the commit's unique ID (SHA). Yours will be different.
HEAD
A pointer: "you are here." Right now it points to your one and only commit.
main
The branch name. It's just a moving pointer to the latest commit.
root-commit
Appears only on the first commit of a repo — the beginning of the history.
Stuck?

"Please tell me who you are" → you skipped Step 4. Go back.

Accidentally ran git commit without -m → you're stuck in the Vim editor. Press Esc, type :wq, Enter. Next time, always use -m "your message".

Phase 2 · GitHub
6

Create an Empty Repo on GitHub

~4 min

Your local repo needs somewhere to live online. You'll create a matching empty repo on GitHub — empty is key, because you already have files locally and you don't want conflicts.

What to do

  1. Open github.com. Sign in if you're not already.
  2. Top-right: click the + icon → New repository.
  3. Fill in:
    • Repository name: same as your local folder (e.g., my-ai-tool)
    • Description: one sentence about your tool
    • Public ✓
    • Do NOT check "Add a README file"
    • Do NOT add a .gitignore or LICENSE
  4. Click Create repository.
  5. GitHub shows you a page titled "Quick setup" with a URL and a few command blocks. Leave this page open. You'll use the URL in the next step.
✓
You're looking at an empty-repo Quick Setup page. The URL near the top ends in .git (e.g., https://github.com/YOU/my-ai-tool.git). Copy that URL — you'll paste it in Step 8.
Stuck?

I accidentally checked "Add a README" → delete this repo (Settings → Danger Zone) and create a new one. Or we'll fix it together in class — tell your instructor.

7

Sign In to GitHub from the Terminal

~6 min

Your terminal doesn't know who you are yet. We'll use GitHub CLI (gh) — one tool, authenticates you for life. If you don't have it installed, pick Option A or Option B below.

Check if you already have it

check if GitHub CLI is installed gh --version gh version 2.x.x (if installed — skip to Authenticate below) 'gh' is not recognized... (if not installed — continue)

If you see "'gh' is not recognized as the name of a cmdlet" or "command not found", GitHub CLI isn't installed yet. Pick one of the two install paths below.

Option A · One-line install (preferred)

Works on most modern Windows 10/11 machines and any Mac with Homebrew.

Windows (PowerShell) winget install --id GitHub.cli Mac (Terminal) brew install gh

Wait 30–60 seconds for the progress bar to finish. You'll see a "Successfully installed" message.

Option B · Download the installer (fallback)

If Option A doesn't work (corporate laptop blocks winget, no Homebrew on Mac, etc.), download the installer manually.

  1. Open cli.github.com in a browser.
  2. Click the big Download for Windows or Download for Mac button — the site detects your OS automatically.
  3. Run the downloaded file:
    • Windows: double-click the .msi file. Accept all defaults (Next → Next → Install → Finish).
    • Mac: double-click the .pkg file. Accept all defaults.
  4. Wait for the installer to complete (usually 30–60 seconds).
⚠ Critical · After installing

Close your terminal / PowerShell completely and open a fresh window. Windows and Mac only load the new gh command into new terminal sessions. If you try gh auth login in the same window where you just installed, you'll see the "not recognized" error again — and wrongly assume the install failed. Close. Reopen. Try gh --version again to confirm.

Authenticate

gh auth login ? What account do you want to log into? GitHub.com ? What is your preferred protocol? HTTPS ? Authenticate Git with your GitHub credentials? Yes ? How would you like to authenticate GitHub CLI? Login with a web browser ! First copy your one-time code: ABCD-1234 Press Enter to open github.com in your browser...

Press Enter. A browser window opens. Paste the one-time code. Click Authorize. Close the browser tab. Back in the terminal, you'll see:

✓ Authentication complete. ✓ Logged in as your-username
✓
Running gh auth status shows "Logged in to github.com as YOUR-USERNAME". Your terminal now has GitHub credentials forever.
Stuck?

"'gh' is not recognized" — AFTER installing → you forgot to close and reopen your terminal. Close every PowerShell / Terminal window, open a fresh one, try gh --version again.

"winget : The term 'winget' is not recognized" → your Windows version predates winget. Use Option B (download the .msi installer) instead.

"brew: command not found" (Mac) → install Homebrew first at brew.sh, OR use Option B (download the .pkg installer) instead.

Corporate laptop blocks everything → ask your instructor for the PAT (Personal Access Token) workaround.

8

Connect Local to Remote with git remote add

~3 min

You have a local repo (your laptop) and an empty remote repo (GitHub). These two commands introduce them to each other and confirm your branch is named main. This is the exact sequence GitHub shows you on the Quick Setup page.

Use the URL you copied in Step 6

"origin" is the conventional name for your main remote git remote add origin https://github.com/YOUR-USERNAME/my-ai-tool.git rename (or confirm) the current branch as "main" git branch -M main confirm the remote was added correctly git remote -v origin https://github.com/YOUR-USERNAME/my-ai-tool.git (fetch) origin https://github.com/YOUR-USERNAME/my-ai-tool.git (push)

"origin" is just a name — by convention, it's the nickname for your primary remote. You could call it anything, but everyone uses origin. The git branch -M main command renames your current branch to main (or confirms it's already called that) — it's safe to run either way.

✓
git remote -v shows a line starting with origin followed by your GitHub repo URL. Nothing has been uploaded yet — you've only established the connection.
◆ Anatomy of the URL
https://github.com/YOU/my-ai-tool.git
https://
Protocol. (SSH uses git@github.com: instead — we'll stick with HTTPS.)
github.com
The host — where the remote lives.
YOU
Your GitHub username (owner of the repo).
my-ai-tool
The repo name.
.git
Conventional suffix marking it as a Git repo URL.
Stuck?

"error: remote origin already exists" → you already added it. Check with git remote -v. If the URL is right, you're fine. If wrong, remove and re-add: git remote remove origin.

9

Push for the First Time

~4 min

This is the moment the files leave your laptop and arrive on the internet.

Run this

-u sets "origin/main" as the default for future pushes git push -u origin main Enumerating objects: 6, done. Counting objects: 100% (6/6), done. Writing objects: 100% (6/6), 12.4 KiB | 2.5 MiB/s, done. To https://github.com/YOUR-USERNAME/my-ai-tool.git * [new branch] main -> main branch 'main' set up to track 'origin/main'.

Verify on GitHub

  1. Go to https://github.com/YOUR-USERNAME/my-ai-tool in your browser.
  2. Refresh.
  3. Your files are there! index.html, README.md, everything.
  4. Click "Commits" at the top of the file list → you see your "Initial commit" with the timestamp.
🌍 Your code is on the internet

Look at your GitHub profile — there is now a green square on your contribution graph. That's a lot more than most people ever get. You'll collect many.

✓
GitHub shows your files. The README renders automatically on the repo page.
Stuck?

"Authentication failed" → Step 7 didn't complete. Re-run gh auth login.

"Updates were rejected" → the remote has commits you don't (probably because you accidentally checked "Add README" in Step 6). Fix: git pull origin main --allow-unrelated-histories, then push again.

Phase 3 · Go Live
10

Deploy with GitHub Pages

~5 min

GitHub Pages is a free service that turns your repo into a public website. In the next five minutes, your AI tool will be live on the internet.

What to do (back in the browser)

  1. On your repo page, click Settings (top menu).
  2. In the left sidebar, click Pages.
  3. Under Build and deployment → Source: select Deploy from a branch.
  4. Under Branch: select main, folder / (root). Click Save.
  5. Wait 30–60 seconds. Refresh the page.
  6. A green banner appears with your URL:
    https://YOUR-USERNAME.github.io/my-ai-tool/
  7. Click the URL. Your AI tool loads in a new tab.
🎉 You're live

Send the URL to yourself by email or text right now so you don't lose it. Send it to someone you love. Put it on your LinkedIn tonight.

✓
Your customized AI tool is live at a public URL. Demo Mode works out of the box — no API key needed.
Stuck?

"My URL shows 404" → first deploy can take up to 2 minutes. Hard-refresh (⌘+⇧+R / Ctrl+F5). Still broken? Confirm Settings → Pages shows the URL as deployed.

"The page loads but GSAP animations don't play" → check your internet connection; GSAP loads from a CDN.

Phase 4 · Iterate
11

Customize Your App with Claude

~25 min

You have a generic AI tool live on the internet. Now make it yours — a Socratic tutor, a negotiation coach, a recipe generator, whatever you'd actually use. There are two paths. Pick whichever fits your vibe.

Path A · Edit what you have (easier, faster)

Keep the app structure; ask Claude to tweak title, personality, and style.

  1. Decide. Take 2 minutes. Pick a concept. Don't overthink it.
  2. Open claude.ai → new chat.
  3. In VS Code, open index.html, select all (⌘+A / Ctrl+A), copy, paste into Claude.
  4. Write a prompt. Example below.
  5. Paste Claude's output back into index.html. Save. Double-click to preview.
Path B · Regenerate a fresh app (ambitious, slower)

Use PROMPT.md — the prompt that authored the app — to generate a completely new one tailored to your vision. This is what your instructor demoed at the start.

  1. Open PROMPT.md in VS Code.
  2. Find the "What the app does" section. Rewrite it to describe your tool in detail (what it does, who it's for, what's special about it).
  3. Optionally change the colors / fonts / animations if you want a visual reset.
  4. Copy everything between the ===== lines.
  5. Paste into a new claude.ai chat. Wait ~60 seconds for Claude to write a fresh HTML file.
  6. Copy Claude's response. Replace your index.html entirely. Save. Double-click to preview.

Path A example prompt

EXAMPLE · PATH A Turn this into a Socratic tutor for university students. Teach by asking guiding questions, never by giving direct answers. Change the title, tagline, system prompt, and the starter suggestion chips. Change the accent color to deep purple. Keep it as a single HTML file. Return the full code.
✓
Your index.html has your tool's new title, personality, and styling. It works in Demo Mode when you double-click to open it locally. It doesn't matter which path you took — both paths produce a customized index.html that commits cleanly in Step 12.
Stuck?

"Claude gave me partial code" → tell Claude: "Return the complete, full HTML file from top to bottom, unabridged."

"The page is broken after Path A" → no problem. In the terminal: git checkout index.html restores the last committed version. Try again with a smaller change.

"Path B generated a broken app" → you probably deleted too much of the prompt. Restore the original PROMPT.md from your ZIP and modify only the "What the app does" section next time.

"I'm 20 minutes in and Path B still isn't working" → switch to Path A. You can always come back to Path B after class.

12

Commit & Push Your Changes

~5 min

You've proven the full loop works once. Now do it again — with your customizations. This is the rhythm you'll repeat for the rest of your career.

The full loop, one more time

see what changed since last commit git status modified: index.html stage the changed file git add index.html commit with a clear, descriptive message git commit -m "Build Socratic tutor with purple theme" [main 4d82f1e] Build Socratic tutor with purple theme 1 file changed, 18 insertions(+), 12 deletions(-) push to GitHub — shorter this time because -u was set in Step 9 git push To https://github.com/YOUR-USERNAME/my-ai-tool.git a7f3c2b..4d82f1e main -> main

Watch it go live

  1. Wait ~30 seconds (GitHub Pages rebuilds automatically).
  2. Open your live URL: https://YOUR-USERNAME.github.io/my-ai-tool/
  3. Hard-refresh (⌘+⇧+R / Ctrl+F5).
  4. Your customizations are live.

Check your history

git log --oneline 4d82f1e (HEAD -> main, origin/main) Build Socratic tutor with purple theme a7f3c2b Initial commit — AI tool starter

Two commits. Your project has a real history now. Every change, forever.

✓
Your live URL shows your customized version. git log shows at least two commits.
